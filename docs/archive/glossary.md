# 術語詞彙表 (Glossary)

本文件定義了專案中使用的核心術語，以確保團隊成員之間有共同的理解和溝通語言。

---

### **維度 (Dimension)**
- **定義**: 一個獨立的、最高層級的協作實體。它可以是「個人維度」（由單一 `User` 擁有）或「組織維度」（由 `Organization` 擁有）。
- **類比**: 類似於 Slack 或 Discord 中的一個「伺服器」，或 GitHub 中的一個「組織」。
- **職責**: 擁有工作區 (`Workspace`)，並作為所有權和計費的邊界。

### **工作區 (Workspace)**
- **定義**: 一個邏輯隔離的專案空間，存在於某個「維度」之內。
- **類比**: 類似於 Slack 中的一個「頻道」，或 Jira 中的一個「專案」。
- **職責**: 包含具體的任務 (`Tasks`)、議題 (`Issues`)、文件 (`Files`) 等資源。它是「能力」被掛載和執行的地方。

### **能力 (Capability)**
- **定義**: 一個獨立的功能模組，可以被「掛載」到工作區中以擴展其功能。
- **範例**: `Tasks`、`QA`、`Daily`、`Finance` 都是獨立的能力。
- **職責**: 封裝一組特定的業務邏輯和 UI，例如任務管理或財務追蹤。能力之間是解耦的，透過 `WorkspaceEventBus` 進行通訊。

### **事件匯流排 (Event Bus)**
- **定義**: 一個在工作區範圍內有效的事件發布/訂閱系統。
- **全名**: `WorkspaceEventBus`。
- **職責**: 實現不同「能力」之間的解耦通訊。例如，當 `Tasks` 能力中的一個任務完成時，它可以發布一個事件，`QA` 能力可以訂閱該事件並將該任務納入待驗證佇列。

### **A-Track / B-Track**
- **A-Track (Acceptance Track)**:
  - **定義**: 任務從創建到最終驗收的「正向流程」。
  - **路徑**: `Todo` → `Doing` → `Completed` (待 QA) → `Verified` (待驗收) → `Accepted` (完成)。
- **B-Track (Backup Track / Anomaly Track)**:
  - **定義**: 當 A-Track 流程中出現問題（如 QA 失敗、驗收失敗）時，用於處理異常和衝突的「備用流程」。
  - **觸發**: 通常會自動創建一個 `Issue`，並將相關任務狀態回滾，交由 B-Track 流程處理。

### **權限共振矩陣 (Permission Resonance Matrix)**
- **定義**: 一個視覺化介面，用於展示「團隊」與「工作區」之間的存取權限映射關係。
- **目的**: 讓管理員可以宏觀地了解哪個團隊可以存取哪個工作區。
- **注意**: 這是一個唯讀的「視圖」。實際的授權操作是在各個工作區的 `Members` 能力中進行的。

### **單向依賴 (One-Way Dependency)**
- **定義**: 本專案的核心架構原則。指程式碼的依賴關係只能是單向的，例如 UI 層可以依賴邏輯層，但邏輯層不能依賴 UI 層。
- **流向**: `app/` → `features/{name}/index.ts` → `shared/*`。
- **目的**: 避免循環依賴，提高程式碼的可測試性和可維護性。

### **kebab-case (短橫線命名法)**
- **定義**: 一種檔案命名規範，使用小寫字母，並用連字號 (`-`) 分隔單字。
- **範例**: `daily-log-card.tsx`。
- **規則**: 這是專案中**強制性**的檔案命名規則，除了 Next.js 的保留檔案外，所有新建檔案都必須遵守。
